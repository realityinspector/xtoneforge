## Cloud workspace vs. local orchestration

GitHub Copilot Workspace and Stoneforge both aim to turn task descriptions into working code, but they take very different paths to get there.

### Copilot Workspace: Issue to PR

Copilot Workspace starts from a GitHub issue, generates a plan (specification, file changes), lets you iterate on the plan, then implements it and creates a pull request. It's a guided, linear workflow tightly integrated with GitHub.

### Stoneforge: Task to merge

Stoneforge takes a broader approach. You create tasks (potentially dozens), define dependencies between them, and the Daemon automatically dispatches them to available agents. Each agent works in an isolated worktree, and Steward agents handle merging and review.

### The parallelism advantage

The biggest difference is scale. Copilot Workspace processes one issue at a time in a guided flow. Stoneforge dispatches as many tasks as you have agents:

```bash
# Plan an entire sprint's worth of work
sf plan create --title "Q1 API Overhaul"

# All of these run in parallel
sf task create --title "Migrate to OpenAPI 3.1" --plan "Q1 API Overhaul"
sf task create --title "Add request validation middleware" --plan "Q1 API Overhaul"
sf task create --title "Implement API versioning" --plan "Q1 API Overhaul"
sf task create --title "Add rate limiting per API key" --plan "Q1 API Overhaul"
sf task create --title "Generate API client SDKs" --plan "Q1 API Overhaul" \
  --depends-on "Migrate to OpenAPI 3.1"
```

### No platform lock-in

Stoneforge works with any git host — GitHub, GitLab, Bitbucket, or self-hosted. It's also agent-agnostic, so you're not tied to any single AI provider. Use the best agent for each task type.
