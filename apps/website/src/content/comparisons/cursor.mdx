## IDE copilot vs. orchestration platform

Cursor and Stoneforge solve fundamentally different problems. Understanding the distinction helps you decide when to use each — or both.

### Cursor: Your AI pair programmer

Cursor is a VS Code fork with deep AI integration. It gives you real-time code completion, inline chat, and the ability to edit code through natural language. It's designed for the developer-in-the-loop workflow: you write code, and AI assists you in real time.

### Stoneforge: Your fleet of autonomous agents

Stoneforge coordinates multiple AI agents working independently in parallel. Instead of one developer with an AI assistant, think of it as dispatching a team of agents that each take a task, work in isolation, and merge their results automatically.

### When you need both

The two tools complement each other naturally:

- **Use Cursor** for interactive work — exploring code, writing features hands-on, debugging issues where you need to see and control every change.
- **Use Stoneforge** for background work — running 5 agents to build 5 features while you focus on architecture decisions in Cursor.

```bash
# While you work interactively in Cursor on the main feature...
# Stoneforge handles the supporting tasks in the background
sf task create --title "Add unit tests for auth module"
sf task create --title "Update API documentation"
sf task create --title "Refactor database queries for performance"
```

### Scale matters

A single Cursor session is powerful for focused work, but it's inherently sequential. Stoneforge breaks that bottleneck by running N agents simultaneously, each with its own worktree, each contributing to different parts of the codebase.
