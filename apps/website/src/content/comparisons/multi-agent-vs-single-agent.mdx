## The shift from copilot to fleet

AI coding tools have evolved through distinct phases. Understanding where multi-agent orchestration fits helps you make better tooling decisions.

### Phase 1: Code completion

Tools like GitHub Copilot and TabNine predict the next line of code. They're fast, low-friction, and work inside your editor. Great for boilerplate and pattern completion.

### Phase 2: AI chat assistants

Cursor, ChatGPT, and similar tools let you have conversations about code. You describe what you want, the AI generates it, and you iterate. More capable than completion, but still one conversation, one task, one developer.

### Phase 3: Autonomous agents

Claude Code, Codex CLI, and OpenCode can work autonomously — reading codebases, running tests, committing code. They complete entire tasks, not just lines of code.

### Phase 4: Multi-agent orchestration

This is where Stoneforge operates. Instead of one autonomous agent, coordinate many. Instead of one task at a time, dispatch an entire sprint. Instead of manual merge, automated conflict resolution.

```bash
# This is the multi-agent difference:
# One command dispatches work across your entire team of agents

sf plan create --title "Sprint 14"

sf task create --title "Implement user authentication" --plan "Sprint 14"
sf task create --title "Build dashboard components" --plan "Sprint 14"
sf task create --title "Add email notification service" --plan "Sprint 14"
sf task create --title "Write integration tests for API" --plan "Sprint 14"
sf task create --title "Update deployment configuration" --plan "Sprint 14" \
  --depends-on "Implement user authentication"

# Agents pick up tasks automatically, work in parallel, merge results
```

### When single-agent is better

Multi-agent isn't always the answer. Single-agent copilots excel at:

- **Exploratory coding** — when you don't know the solution yet and need to iterate interactively
- **Debugging** — when you need to trace through code step by step with AI assistance
- **Learning** — when the goal is understanding, not output
- **Small, quick edits** — when orchestration overhead exceeds the task complexity

### When multi-agent wins

Multi-agent orchestration dominates when:

- **You have 3+ parallelizable tasks** — the throughput advantage kicks in fast
- **Tasks are well-defined** — clear acceptance criteria let agents work autonomously
- **The codebase is large** — parallel agents can touch different areas without conflict
- **You need reproducibility** — each agent works in isolation, making results deterministic
- **You're planning sprints** — dispatch an entire backlog and review results instead of coding each task yourself
