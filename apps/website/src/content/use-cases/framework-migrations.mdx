Stoneforge turns daunting framework migrations into a parallel, incremental operation. Define the transformation pattern once, and let agents apply it across your entire codebase simultaneously.

### Plan the migration graph

Migrations have dependencies. You can't migrate the components before migrating the shared utilities they depend on. Stoneforge's [Director agent](https://docs.stoneforge.ai/core-concepts/agents/) analyzes your import graph and creates a dependency-ordered plan.

```bash
# Describe the migration
sf plan create --title "Migrate from JavaScript to TypeScript" \
  --description "Convert all .js files to .ts/.tsx. Add type annotations. Update tsconfig. Fix type errors. Maintain all existing tests."

# The Director creates an ordered plan:
# Phase 1 (parallel): Add tsconfig.json, install dependencies
# Phase 2 (parallel): Migrate src/types/ and src/utils/ (no deps)
# Phase 3 (parallel): Migrate src/services/ (depends on types)
# Phase 4 (parallel): Migrate src/components/ (depends on utils)
# Phase 5 (parallel): Migrate src/pages/ (depends on components)
# Phase 6: Update build config and CI pipeline
```

### Consistent transformations at scale

The key to a successful migration is consistency. Every file should be transformed the same way. Stoneforge enforces this through [workspace prompts](https://docs.stoneforge.ai/guides/custom-prompts/) that define the exact transformation rules.

```bash
# .stoneforge/prompts/worker.md:
# "TypeScript migration rules:
#  - Use strict mode, no implicit any
#  - Prefer interface over type for object shapes
#  - Use unknown instead of any for external data
#  - Add return types to all exported functions
#  - Convert require() to import statements
#  - Preserve existing JSDoc as TSDoc"
```

### Incremental, not big-bang

Each phase produces small, mergeable PRs. Your codebase transitions from the old framework to the new one incrementally — no long-lived migration branch that goes stale. If something goes wrong, you roll back one PR, not the entire migration.

### Parallel execution within phases

Within each phase, agents work in parallel. If Phase 3 has 15 service files to migrate, three agents can handle five files each — finishing in a third of the time. The dependency system ensures phases execute in the right order while maximizing parallelism within each phase.

```bash
# Start the migration
sf serve

# Dashboard shows parallel progress:
# Worker-1: Migrating src/services/auth.js → auth.ts (45%)
# Worker-2: Migrating src/services/billing.js → billing.ts (72%)
# Worker-3: Migrating src/services/users.js → users.ts (100% ✓)
```

### Test-driven migration

Every migrated file is tested before merge. Agents run existing tests against the migrated code and fix any failures. For TypeScript migrations, agents also ensure zero type errors with `tsc --noEmit` before submitting the merge request.
